# Lab 01: Exploring Memory Layout with `.map` File and Heap Debugging

## 🎯 Objectives

By the end of this lab, students should be able to:

- Generate a `.map` file in Visual Studio.
- Identify memory sections such as `.text`, `.data`, and `.bss`.
- Distinguish between **stack**, **heap**, and **global memory**.
- Observe memory addresses during debugging.

---

# Background

Modern programs use several memory regions during execution.

| Memory Region | Purpose |
|---------------|--------|
| `.text` | Stores compiled machine instructions |
| `.data` | Stores initialized global/static variables |
| `.bss` | Stores uninitialized global/static variables |
| Stack | Stores local variables and function frames |
| Heap | Stores dynamically allocated memory |

The **`.map` file** generated by the linker shows where compiled symbols are placed in memory.

---

# Procedure

## Step 1 — Create a Program

Create a new **C++ Console Application** and use the following code.

```cpp
#include <iostream>
using namespace std;

int global_var = 42;     // .data section
int global_uninit;       // .bss section

void demoFunction() {
    cout << "Inside demoFunction" << endl;
}

int main() {
    int local_var = 5;           // stack
    int* heap_var = new int(99); // heap

    demoFunction();

    cout << "Heap address: " << heap_var << endl;

    delete heap_var;
    return 0;
}
```

---

## Step 2 — Enable `.map` File Generation

1. Right-click the project → **Properties**
2. Go to **Linker → Debugging**
3. Set **Generate Map File = Yes**
4. Optionally specify a name (example: `lab1.map`)
5. Rebuild the project

The `.map` file will appear in the **build output directory**.

---

## Step 3 — Inspect the `.map` File

Open the generated `.map` file in a text editor.

Locate entries corresponding to:

- `.text`
- `.data`
- `.bss`

Example entries may look similar to:

```
.text       0001:00000000       demoFunction
.data       0002:00000000       global_var
.bss        0003:00000000       global_uninit
```

Note:

Heap and stack are **runtime structures** and therefore **do not appear in the map file**.

---

## Step 4 — Debug Stack vs Heap

1. Start debugging (**F5**)
2. Set a **breakpoint at `main()`**
3. Step through the program
4. Observe addresses of:
   - `local_var`
   - `heap_var`

Typical observations:

- Stack addresses are located in the **stack frame region**
- Heap addresses come from the **heap allocator region**

---

# Expected Observations

| Variable | Memory Region |
|--------|-------------|
| `global_var` | `.data` |
| `global_uninit` | `.bss` |
| `local_var` | stack |
| `heap_var` | heap |

---

# Analysis Questions

1. Why do **global variables appear in the `.map` file** but heap variables do not?
2. What is the difference between **`.data` and `.bss` sections**?
3. Why are **stack addresses usually close together**?
4. What role does the **memory allocator** play in heap management?

---

# Conclusion

This experiment demonstrates how a compiled program is organized into multiple memory regions.  
The `.map` file provides visibility into **compile-time memory placement**, while debugging allows observation of **runtime memory allocation**.

Together, these tools help understand the internal structure of program execution.
